# Judgment-Driven Development

**Sprint Kit 설계 철학**

> jdd-sprint-kit은 제품 전문가(product expert) — 개발자가 아닌 — 가
> AI를 활용하여 자신의 판단을 소프트웨어 산출물로 실현할 수 있게 하는 도구를 지향한다.
> 사용자는 코드를 모르지만, 고객이 무엇을 원하는지는 안다.
> 사용자가 하는 일은 단 세 가지다:
> 무엇을 만들지 결정하고, 결과물이 맞는지 확인하고, 틀리면 이유를 말한다.
> 나머지는 전부 AI가 처리하며,
> 라이브 서비스 개발 속도를 극적으로 가속하는 것이 목표이다.
>
> 하나의 원칙: **인간의 판단만이 유일한 영속 자산이다. 모든 AI 산출물은 일회용이며 재생성 가능하다.**
>
> 기반 관점과 가설 체계: [`docs/translation-ontology.md`](translation-ontology.md)

## 배경: BMad Method와 Sprint Kit

### BMad Method — "신뢰할 수 있는 AI 협업"

BMad Method는 AI를 퍼실리테이터(facilitator, 진행 촉진자)로 활용하는 하향식 기획 프레임워크이다.
12단계 step-file 아키텍처를 통해 사용자가 매 단계 의사결정에 참여하며,
고품질 기획 산출물(Product Brief, PRD, Architecture, Epics)을 생산한다.

핵심 가치: **탐색과 발견을 통한 빈틈 없는 요구사항 정의**

### Sprint Kit — "산출물로 대화한다"

Sprint Kit은 BMad Method를 활용하여 기획 산출물을 생성하는 실행 계층이다.
사전 입력(회의록, 참고 자료)과 기존 시스템 맥락을 자동으로 수집하고,
AI가 기획 산출물을 생성하면, 제품 전문가가 핵심 시점에서 판단한다.

핵심 가치: **구체적인 산출물에 대한 전문가 판단을 통한 빠른 실현**

### 관계

Sprint Kit은 현재 BMad Method를 활용하며, 에이전트 경로와 포맷 가이드에서 BMad에 의존한다. 그러나 Sprint Kit의 논리적 핵심(번역, 델타, 판단)은 BMad에 의존하지 않는다. **기존 인간 지식의 형태**에 따라
서로 다른 루트를 제공하며, 동일한 산출물 포맷으로 수렴하여 같은 실행 파이프라인으로 흘러간다.

```
BMad Method (현재 활용 중)
  ├─ Agent Framework (Mary, John, Winston, Sally, Bob...)
  ├─ Workflow Engine (step-file architecture)
  ├─ Facilitation Patterns (A/P/C menu, party mode)
  └─ Artifact Formats (PRD, Architecture, Epics)

Sprint Kit (BMad를 활용하는 실행 계층)
  ├─ Input Layer: 사전 입력 처리 + brownfield 자동 수집
  ├─ Generation Layer: BMad 에이전트 자동 오케스트레이션
  ├─ Judgment Layer: Customer-Lens JP1/JP2
  ├─ Execution Layer: Specs → Deliverables → Prototype
  └─ Regeneration: Apply-fix + propagation / regeneration pipeline
```

> BMad 의존성 분리 로드맵: [`reviews/translation-ontology-roadmap.md`](reviews/translation-ontology-roadmap.md) §1

## Design Judgments (설계 판단)

### Artifacts as Medium (매체로서의 산출물)

> 추상적 질문은 부정확한 답을 낳는다. 구체적 산출물은 정확한 반응을 이끌어낸다 — 가장 빠르고 정확한 입력이다.

사람은 추상적 질문에 부정확하게 답하지만, 구체적 산출물에는 정확하게 반응한다.

```
추상적 질문:     "검색 기능에서 가장 중요한 것은?"      → 부정확한 답변
구체적 산출물:   "이 검색 화면이 맞습니까?"              → 정확한 판단

추상적 질문:     "어떤 API가 필요한가?"                  → 모호한 답변
구체적 산출물:   "이 5개 엔드포인트로 충분한가?"         → 명확한 판단
```

이 판단은 BMad Method의 "Facilitation Over Generation(생성보다 촉진)"을 확장한 것이다.
BMad가 대화를 통해 발견을 촉진하는 반면,
Sprint Kit은 **산출물을 통해 판단을 촉진한다.**

두 접근 모두 인간의 전문성을 이끌어낸다.
어느 쪽이 더 효과적인지는 사용자의 맥락과 상황에 따라 다르다.

### Input Reduces Cycles (입력이 순환을 줄인다)

> 사전 입력(회의록, 참고 자료, 기존 시스템 맥락)은 첫 번째 생성의 품질을 높여
> 재생성 횟수를 줄인다. 좋은 입력은 여러 번의 판단보다 효율적이다.

AI 생성 비용은 0이 아니다. 재생성 횟수가 늘수록 인간의 판단 시간이 누적된다.

```
총 비용 = (사전 입력 비용) + (생성 비용 × 생성 횟수) + (판단 비용 × 판단 횟수)

풍부한 사전 입력:    생성 횟수 ↓, 판단 횟수 ↓  → 총 비용 ↓
사전 입력 없음:      생성 횟수 ↑, 판단 횟수 ↑  → 총 비용 ↑
```

Sprint Kit에서의 실현 방식:
- `specs/{feature}/inputs/`: 회의록, 참고 자료, 기존 문서를 넣는 곳
- Brownfield Scanner: 기존 시스템 맥락을 추가 입력으로 자동 수집
- Brief: 사용자가 제공하는 최소한의 방향 설정

이 세 가지가 결합되어 AI의 첫 번째 생성이 "대충 맞는" 수준이 아니라 "검토할 가치가 있는" 수준에 도달한다.

**실제 사례**: 제품 팀이 2시간짜리 킥오프 회의를 하고
회의록이 문서로 존재한다면, 그 회의록에는 이미
BMad 12단계 인터뷰가 발견할 정보의 대부분이 담겨 있다.
이를 inputs/에 넣기만 해도 첫 번째 PRD 생성 품질이 크게 올라간다.

### Regeneration Over Modification (수정보다 재생성)

> 재생성이 기본값이지, 수정이 아니다. 모든 AI 산출물은 일회용이지만,
> 재생성 비용이 0이 아니므로 Input과 Judgment가 재생성 횟수를 최소화한다.

AI 이전의 수정(패칭) 방식:
```
문서 v1 → 피드백 → v1 부분 수정 → v1.1 → 피드백 → 부분 수정 → v1.2
문제: 누적된 패치가 문서의 일관성을 깨뜨린다
```

AI 시대의 재생성 방식:
```
문서 v1 → 피드백 → 피드백을 반영한 완전히 새로운 v2 → 피드백 → v3
장점: 매번 일관된 결과물
전제: 인간의 판단(피드백)이 축적되어 다음 생성에 투입된다
```

**인간의 판단만이 유일한 영속 자산이다. 나머지는 전부 재생성 가능하다.**

**출처 기반 재생성 범위(provenance-based regeneration scope)**: 재생성 범위는 피드백의 출처에 의해 정확히 결정된다:
- 사용자의 원본 입력에 이미 존재 → 번역/생성 오류 → 해당 단계만 재생성
- 사용자 입력에 없지만 추론 가능 → AI 추론 누락 → 해당 단계 + 하위 단계 재생성
- 사용자 입력에 완전히 부재 — 새로운 요구사항 → Brief 수정 후 파이프라인 재시작

출처 기반 분류는 보다 정밀한 재생성 비용 추정을 가능하게 한다.

Sprint Kit에서의 실현 방식:
- Circuit Breaker: 비상 모드가 아닌 정상적인 재생성 트리거
- JP Comment → Regeneration: 피드백 크기에 따라 재생성 범위가 동적으로 결정
- JP Comment → Apply-fix + propagation: 작은 피드백을 기존 산출물 내에서 양방향 전파 + Scope Gate 검증
- 어떤 방식이든 사용자는 비용을 확인한 후 선택한다 (비용 투명성)

**실질적 균형**: 재생성 비용이 0이 아니므로,
Input Reduces Cycles가 재생성 횟수를 줄이고,
Customer-Lens Judgment Points가 적절한 시점에 방향을 교정하여
불필요한 재생성을 방지한다.

### Customer-Lens Judgment Points (고객 관점 판단 시점)

> 인간의 개입은 "고객에게 어떤 제품이 제공될 것인가"를
> 판단할 수 있는 시점에 배치된다.
> 판단은 항상 구체적인 산출물 위에서 이루어진다.

제품 전문가의 정의:
- ~~개발자의 반대로서의 "비개발자"~~
- **고객 전문가 — 산출물이 어떤 모습이어야 하는지 판단하기에 가장 적합한 사람**

체크포인트는 이 정의에서 도출된다.

이상적으로는, Brief 입력에서 곧바로 프로토타입(JP2)까지 가서 사용자가 산출물만 판단하면 된다. 그러나 JP1에는 JP2가 대체할 수 없는 고유한 **방향 검증** 기능이 있다: 프로토타입에 없는 누락 시나리오 발견, 요구사항 수준의 우선순위 판단, 고객 여정과의 정합성 확인은 JP2에서 수행할 수 없다. 또한 현재 AI 속도에서는 JP2까지 한 번에 도달하는 데 수십 분이 걸리므로, JP1은 고유한 방향 검증 역할과 실용적 보완 역할을 동시에 수행한다. AI가 충분히 빨라지더라도 JP1의 형태는 변할 수 있지만(예: 요구사항 체크리스트를 프로토타입과 함께 제시), 방향 검증 기능 자체는 유지될 수 있다.

**JP1 (Judgment Point 1): "고객에게 적합한 제품인가?"**
```
대상:     요구사항, 사용자 시나리오, 기능 범위, 우선순위
형식:     PRD 핵심 내용을 고객 여정 서사로 제시
          "고객이 상황 A에서 B를 하려 할 때, 시스템은 C를 제공한다."
제품 전문가의 판단:
  - "이 시나리오가 실제 고객 상황과 맞는가?"
  - "빠진 시나리오는 없는가?"
  - "우선순위가 맞는가?"
응답: Confirm / Comment
```

**JP2 (Judgment Point 2): "고객이 원하는 경험인가?"**
```
대상:     프로토타입, 화면 흐름, 인터랙션
형식:     동작하는 프로토타입 + 핵심 시나리오 가이드
          "시나리오 1을 시도하세요: 로그인 → 대시보드 → 기능 X"
제품 전문가의 판단:
  - "이 화면 배치가 고객에게 자연스러운가?"
  - "기능이 고객이 기대하는 대로 동작하는가?"
  - "빠진 화면이나 흐름은 없는가?"
응답: Confirm / Comment
```

**Comment — 피드백이 방향을 교정한다:**

Comment가 선택되면, 시스템이 피드백의 영향 범위를 분석하고
두 가지 옵션을 비용과 함께 제시한다: apply-fix + propagation 대 regeneration.
사용자는 비용을 확인한 후 선택한다.

```
JP1/JP2 → Comment → 피드백 입력
  → 시스템: 영향 분석
  → [Apply-fix + propagation] N개 파일, ~M분, Scope Gate 검증 포함
  → [Regeneration] Phase X부터, ~M분
  → 사용자 선택 → 실행 → JP로 복귀
```

재생성 범위는 피드백 크기에 따라 동적으로 결정된다.
작은 피드백은 Deliverables만 재생성하고, 큰 피드백은 PRD부터 시작하며,
방향 전환은 Brief 수정과 Sprint 재시작을 유발한다.
JP2에서 요구사항 자체가 잘못되었음이 드러나면, 재생성 범위가
자연스럽게 JP1 이전으로 확장된다 — 이는 실패가 아니라
**구체적 산출물이 촉진한 정상적인 발견이다.**

**구조적 보조 — BMad 12단계의 가치를 체크리스트로 압축:**

제품 전문가가 JP1에서 판단할 때, BMad 12단계의 발견 사항이
구조적 체크리스트로 제공된다:
```
□ 모든 핵심 사용자 유형이 포함되었는가?
□ 엣지 케이스 시나리오가 고려되었는가?
□ 기존 기능과의 관계가 명확한가?
□ 성공 지표가 측정 가능한가?
```

이 체크리스트는 상향식의 약점(빈틈 발견의 어려움)을
하향식의 강점(구조적 탐색)으로 보완한다.

### Knowledge Shape Determines Route (지식의 형태가 루트를 결정한다)

> 루트는 인간의 지식이 어떤 형태로 존재하느냐에 따라 결정된다.

```
인간 지식의 형태                        최적 루트
──────────────────────────────────────────────────────
A. 풍부한 비정형 맥락                  상향식 (Sprint Kit Auto Sprint)
   (회의록, 데이터, 경험)              AI가 구조화 → 인간이 판단 → 재생성

B. 미개척 영역                         하향식 (BMad 12-step)
   (새로운 시장, 새로운 문제)          AI가 질문 → 인간이 발견 → AI가 구조화

C. 이미 구조화된 산출물                직접 실행 (Sprint Kit /specs)
   (기존 PRD, 완성된 스펙)             곧바로 실행 파이프라인으로
```

**선택은 사용자 맥락에 따른다. 시스템은 루트를 강제하지 않는다.**

실제 사용 패턴:
| 사용자 상황 | 입력 형태 | 최적 루트 | 진입점 |
|-------------|-----------|-----------|--------|
| 킥오프 회의 후 PM | 회의록 + 참고 자료 | 상향식 | `/sprint feature-name` |
| 신규 제품을 구상 중인 창업자 | 아이디어만 | 하향식 | `/create-product-brief` |
| 목업 기반으로 실행하는 디자이너 | Figma URL + 설명 | 상향식 | `/sprint feature-name` |
| 기존 PRD를 실행 | 완성된 PRD | 직접 실행 | `/specs feature-name` |
| 범위 확인용 빠른 프로토타입 | 한 줄 Brief | 상향식 | `/sprint "description"` |

**하향식과 상향식은 "깊이"가 다른 것이 아니다:**
- 하향식의 깊이: 문제 정의의 정밀함 — "정확히 무엇을 만들어야 하는가"
- 상향식의 깊이: 판단의 정확함 — "이것이 맞는 것인가"
- 둘 다 깊다. 깊은 곳이 다를 뿐이다.

### Auto-Context, Human-Judgment (자동 맥락, 인간 판단)

> 기술적 맥락 수집은 AI가 자동화한다.
> 인간에게는 고객 영향으로 번역하여 판단만 요청한다.

브라운필드(brownfield, 기존 시스템이 있는) 환경에서 기존 시스템과의 관계를 매핑하는 것은
제품 전문가의 일이 아니다. AI가 자동으로 수집하고,
판단에 적합한 형태로 제품 전문가에게 제시한다.

```
AI가 수집하는 것:                      인간에게 제시되는 것:
"기존 API /api/v1/tutors에             "기존 '튜터 관리' 기능이
 GET, POST, DELETE 엔드포인트가         영향을 받습니다. 현재 튜터 목록
 있음. TutorService 클래스에             화면에 새로운 '차단' 기능을
 blockTutor 메서드가 없음.              추가하면 기존 사용자 경험이
 DB에 tutor_block_list 테이블 없음"     변경됩니다. 승인하시겠습니까?"
```

Sprint Kit에서의 실현 방식:
- Brownfield Scanner: MCP 서버, document-project, 로컬 코드베이스에서 자동 수집
- brownfield-context.md: L1(도메인) → L2(행위) → L3(컴포넌트) → L4(코드)의 계층 구조
- JP1/JP2에서: 기술적 brownfield 데이터를 고객 영향으로 번역하여 제시

## 실제 제품 팀 워크플로와의 매핑

Sprint Kit은 실제 제품 팀 워크플로에서 AI가 처리할 수 있는 부분을 대체하면서,
인간이 뛰어난 부분은 보존한다.

```
실제 제품 팀                            Sprint Kit 대응
──────────────────────────────────────────────────────────────
1. 킥오프 회의 (2시간)                 → 회의록을 inputs/에 저장 (~0분)
2. 누군가 PRD 작성 (1일)              → ██ AI가 PRD 생성 ██ (~5분)
3. PRD 리뷰 회의 (1시간)              → JP1: PRD 판단 (~10분)
4. PRD 수정 (반나절)                   → ██ 필요시 AI 재생성 ██ (~3분)
5. 디자인 → 프로토타입 (1주일)        → ██ AI가 프로토타입 생성 ██ (~10분)
6. 프로토타입 리뷰 (1시간)            → JP2: 프로토타입 판단 (~15분)
7. 수정 → 최종 승인 (수일)            → ██ 필요시 AI 재생성 ██ (~10분)

인간 소요 시간: ~25분 (이전: 4시간 30분 + 수일의 대기)
```

**AI가 대체하는 것:** 구조화, 작성, 구현 (인간이 상대적으로 느린 작업)
**인간이 유지하는 것:** 맥락 제공, 판단, 방향 설정 (인간이 훨씬 정확한 작업)

## AI 시대의 방법론 진화

### Waterfall → Agile의 교훈

하향식(Waterfall) 개발은 "빈틈 없는 사전 정의"를 추구했으나,
실제로는 빈틈을 0으로 줄일 수 없었다. 이 한계를 인정하고
반복을 통해 보완한 것이 Agile의 핵심 통찰이었다.

### Agile → Judgment-Driven 전환

상향식(Judgment-Driven) 개발은 "AI가 만들고, 인간이 판단한다"를 추구하지만,
실제로는 AI 생성 비용과 시간이 0에 도달할 수 없다.
이 한계를 인정하고, **사전 입력으로 첫 생성 품질을 높이며
적절한 판단 시점에서 방향을 교정하는 것**이 Sprint Kit의 핵심 설계이다.

```
Waterfall:  빈틈 없이 정의 → 한 번에 구현 (이상)
            빈틈 발견 → 높은 되돌림 비용 (현실)

Agile:      반복을 통한 점진적 개선 (해결책)
            하지만 매 반복마다 구현 비용 발생

JDD:        AI가 빠르게 생성 → 인간이 판단 → 재생성 (이상)
            재생성 비용 ≠ 0, 판단에도 시간 소요 (현실)
            → 입력 품질 ↑ + 판단 시점 최적화 = 최소 재생성 (해결책)
```

**이것은 실제로 많은 사람이 사용하는 서비스의 핵심이다.**
이론적 이상이 아니라, 실제 제품 조직에서 작동하는 균형점을 찾는 것이다.

### Delta-Driven Design

Sprint Kit의 3-pass 패턴(Answer Discovery → Translation & Delta Extraction → Delta Execution)은 파이프라인의 목적을 재정의한다: 스펙을 생성하는 것이 아니라, **현재 시스템(brownfield)과 목표 상태(사용자가 검증한 프로토타입) 사이의 델타를 정의하는 것**이다.

프로토타입은 시스템의 실제 사용자 문법으로 표현된 목표 상태이다 — 인간 사용자에게는 시각적 UI, 서비스 소비자에게는 API 목업, AI 소비자에게는 구조화된 문서가 된다. Crystallize는 이 목표를 개방형 추상화가 아닌 규칙 기반 매핑을 사용하여 개발 문법으로 번역한다. 번역된 목표와 brownfield 기준선 사이의 델타가 구축해야 할 것이다.

스펙 완전성(spec completeness)은 AI 비결정성(non-determinism)을 제어한다: 스펙이 완전할수록 더 일관된 출력을 생산한다. Sprint Kit은 트레이드오프를 수용한다 — 계약이 중요한 곳(API, DB 스키마)에서는 높은 명세, 변이가 수용 가능한 곳(UI 레이아웃, 내부 명명)에서는 낮은 명세 — 그리고 계약 테스트(contract testing)와 BDD를 사용하여 변이와 무관하게 기능적 정확성을 검증한다.

> 전체 이론, 핵심 원칙(FP1-FP6), 설계 판단(DJ1-DJ10), 번역 규칙, 방법론 비교: [`docs/delta-driven-design.md`](delta-driven-design.md)

## 부록: 설계 논의 과정의 주요 인사이트

### "비개발자" 프레이밍의 문제

초기 논의에서 Sprint Kit의 대상 사용자는 "비개발자"로 정의되었다.
이는 감산적 정의 — 개발자의 반대 — 로서
이 사용자들의 실제 전문성을 포착하지 못한다.

**수정된 정의**: "고객 전문가 — 산출물이 어떤 모습이어야 하는지
판단하기에 가장 적합한 사람" — 즉, 제품 전문가이다.

이 재정의가 체크포인트 설계에 미친 영향:
- 이전: 기술적 품질 게이트 (Brief 매핑 %, Scope Gate 결과)
- 이후: 고객 관점 판단 시점 (고객 여정 서사, 경험 검증)

### "깊이 vs 빠름"이 아닌 "하향식 vs 상향식"

초기 논의에서 루트는 "탐색 깊이"로 분류되었다 (깊은 기획 vs 빠른 실행).
이 프레이밍은 전형적인 개발자 관점으로, 상향식이 "얕다"는 함의를 갖는다.

**수정된 프레이밍**: 두 루트 모두 깊다. 깊은 영역이 다를 뿐이다.
- 하향식: 문제 정의의 정밀함에서 깊다
- 상향식: 산출물에 대한 판단의 정확함에서 깊다

### Bridge 개념의 해소

초기 설계에서는 BMad 산출물을 Sprint Kit으로 이관하는 "Bridge" 커맨드를 계획했다.
논의 결과 이 개념이 불필요함이 드러났다:

1. 두 시스템은 이미 동일한 산출물 포맷을 공유한다 (동일한 prd-format-guide)
2. Sprint Kit의 `/specs`는 planning-artifacts 경로만 해석하면 연결된다
3. Bridge는 "두 시스템 간의 번역기"이지만, 동일한 포맷을 공유하므로 번역이 불필요하다

**결론**: Sprint Kit은 BMad와 별개의 시스템이 아니라 BMad를 활용하는 실행 계층이므로,
연결 지점은 커맨드가 아니라 파일 포맷 계약(planning-artifacts/ 디렉토리)이다.

### "재생성 비용 ≈ 0" 가정의 현실적 교정

초기 논의는 "AI 재생성 비용이 거의 0"이라는 전제 위에서 진행되었다.
현실 확인:

```
전체 Auto Sprint 사이클: 5–15분
PRD 재생성: 2–5분
프로토타입 재생성: 5–10분
```

비용이 0이 아니다. 재생성 횟수가 늘수록 인간의 판단 시간이 누적된다.
이 현실적 제약이 Input Reduces Cycles가 존재하는 이유이다.

**실질적 균형점**: 사전 입력과 brownfield 맥락으로
첫 생성 품질을 높여 재생성을 1~2회로 줄이는 것이 목표이다.

### Regeneration Over Modification의 실무적 교정 — Apply-Fix가 허용되는 경우

"재생성이 수정보다 우선"이 기본값이다. 산출물은 일회용이며,
피드백을 반영한 새로운 생성이 일관성을 보장한다.

실제 Sprint 실행 중 발견된 경계 조건:
파이프라인에서 산출물은 독립적인 일회용품이 아니라 **중간 상태**이다.
동일한 정보(예: 데이터 모델)가 PRD, Architecture, Design,
API Spec, Prototype 등 여러 곳에 다른 형식으로 존재한다.
작은 변경에 전체 재생성을 적용하면 비용 대비 효과가 역전된다.

```
전제:         산출물은 독립적인 일회용품 → 재생성이 항상 효율적
실제 발견:    산출물은 상호 의존하는 상태 → 작은 변경에 재생성은 비효율적
교정:         apply-fix + propagation 옵션 추가 + 일관성을 위한 Scope Gate 검증
```

apply-fix가 안전한 이유: 재생성을 기본값으로 두는 근거는
"수정이 일관성을 깨뜨린다"는 것이다.
Scope Gate가 수정 후 산출물 간 일관성을 검증하므로,
일관성 붕괴의 위험이 구조적으로 차단된다.

**재생성이 기본값이다.** Apply-fix는
시스템이 비용과 함께 제시하는 옵션이다. 사용자는 비용을 확인한 후 결정한다.
프로토콜(jdd-sprint-protocol.md)에 구현되어 있다.
