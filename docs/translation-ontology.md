## 1. 근본 관점

**"Sprint Kit의 핵심 연산(Crystallize)은 사용자 문법에서 개발 문법으로의 번역이다. 번역의 방향은 사람만이 정할 수 있다."**

이 선언은 두 부분으로 분해된다:

| 부분 | 성격 | 핵심 |
|------|------|------|
| **A1 — 존재론적**: "핵심 연산은 번역이다" | Sprint Kit이 *무엇을 하는가* | 사용자 문법과 개발 문법 사이에 규칙 기반 매핑이 존재하며, Crystallize는 이 매핑을 실행한다 |
| **A2 — 가치론적**: "방향은 사람만이 정한다" | Sprint Kit이 *왜 사람을 필요로 하는가* | 번역 대상(무엇을 번역할 것인가)의 결정은 기계적으로 도출되지 않는다 |

A1에서 Delta-Driven Design이 도출된다 — 번역이 핵심 연산이라면, 델타는 번역의 자연스러운 산출물이다.
A2에서 Judgment-Driven Development가 도출된다 — 방향을 사람이 정한다면, 판단 시점(JP)이 필수적이다.

### 범위 한정

"번역" 프레이밍은 Crystallize에 가장 직접적으로 적용된다. Sprint Kit의 전체 파이프라인에는 번역이 아닌 활동이 포함된다:

| Phase | 활동 | 번역과의 관계 |
|-------|------|--------------|
| Phase 1 (답 발견) | 솔루션 공간 탐색, PRD 생성, Architecture 결정 | 번역이 아님 — **번역 대상을 결정하는 탐색** |
| Phase 2 (Crystallize) | 사용자 문법 → 개발 문법 변환, 델타 추출 | **번역 자체** |
| Phase 3 (실행) | 델타 구현, 코드 생성 | 번역이 아님 — **번역 결과의 실현** |

따라서 "소프트웨어는 번역이다"가 아니라 **"Sprint Kit의 핵심 연산은 번역이다"**가 정확한 선언이다. Phase 1(탐색)과 Phase 3(실행)은 번역의 전후 맥락이지, 번역 자체가 아니다.

## 2. 전제 체계

근본 관점은 공리(axiom)가 아니다. 여러 숨겨진 전제 위에 서 있으며, 이 전제들이 깨지면 근본 관점의 유효성도 변한다. 이 전제들을 명시한다.

### 2.1 근본 관점 (A1 + A2)

| ID | 선언 | 성격 |
|----|------|------|
| **A1** | Sprint Kit의 핵심 연산은 사용자 문법에서 개발 문법으로의 번역이다 | 존재론적 |
| **A2** | 번역의 방향(무엇을 번역할 것인가)은 사람만이 정할 수 있다 | 가치론적 |

### 2.2 보조 전제 (H1-H4)

A1과 A2가 성립하기 위해 필요한 구조적 전제들이다. 이것들은 Sprint Kit의 설계가 올바르게 작동하기 위한 조건이다.

| ID | 전제 | 근거 | 깨지는 조건 |
|----|------|------|------------|
| **H1** | 사용자 문법과 개발 문법은 분리 가능하다 | 시스템의 사용자와 구현자는 서로 다른 표현 체계를 사용한다 | 사용자가 곧 개발자인 경우 (개발자 도구, CLI 등) |
| **H2** | 두 문법 사이에 결정론적 매핑 규칙이 존재한다 | Translation Rules 테이블이 유한하고 열거 가능하다 | 매핑이 본질적으로 창발적이어서 규칙화할 수 없는 경우 |
| **H3** | 사용자는 자신의 문법으로만 정확성을 검증할 수 있다 | 프로덕트 전문가는 프로토타입을 통해 판단하지, 명세를 통해 판단하지 않는다 | 사용자가 개발 문법을 구사할 수 있는 경우 |
| **H4** | 번역에서 소실되는 정보(캐리포워드)는 별도 출처에서 복원 가능하다 | PRD, Architecture, Brownfield에서 NFR, 보안, 마이그레이션 등을 주입한다 | 캐리포워드 출처 자체가 불완전하거나 접근 불가한 경우 |

### 2.3 경험적 전제 (H5-H7)

Sprint Kit의 경제적 원칙이 성립하기 위한 경험적 조건들이다. 이것들은 현재 기술 수준에서 관찰되는 사실이며, 기술 발전에 따라 변할 수 있다.

| ID | 전제 | 현재 상태 | 변동 가능성 |
|----|------|----------|------------|
| **H5** | AI 생성 품질이 "리뷰할 만한 수준"이다 | 참 (2024-2026 LLM 기준) | AI 퇴보 시 깨짐 |
| **H6** | 사전 입력이 실제로 생성 품질을 향상시킨다 | 관찰됨 (Grade A vs C Brief 차이) | 입력 무관 모델 등장 시 약화 |
| **H7** | AI 재생성 비용이 "감당 가능"하다 (5-15분/사이클) | 참 (현재 기준) | 비용 급증 시 깨짐; 비용 급감 시 JP1 불필요 |

H7이 특히 중요하다. 재생성 비용이 충분히 낮아지면(≈ 0), JP1의 존재 이유가 사라진다 — Brief에서 바로 JP2로 직행할 수 있다. 반대로 비용이 급증하면, "소모품 취급"이 불가능해져 패치 기반 수정으로 퇴행한다.

### 전제 구분의 의미

| 구분 | 전제 | 의존하는 원칙 |
|------|------|-------------|
| **논리적 필연** | A1, A2, H1, H2, H3, H4 | DDD, JDD, FP1-FP6 |
| **기술 조건 의존** | H5, H6, H7 | 비용 공식, JP1 존재, 재생성 기본 원칙 |

기술 조건 의존 전제가 깨져도 논리적 구조(두 문법, 번역, 판단 필요성)는 유효하다. 다만 실현 방식이 달라진다.

## 3. 도출 구조 (5-Layer)

근본 관점에서 Sprint Kit의 원칙, 설계 판단, 구현에 이르는 도출 경로를 5개 층위로 정리한다.

| Layer | 이름 | 내용 | 예시 |
|-------|------|------|------|
| **L1** | 근본 관점 | A1 + A2 | "핵심 연산은 번역이다. 방향은 사람이 정한다." |
| **L2** | 전제 | H1-H7 | "두 문법은 분리 가능하다", "AI 생성이 리뷰 가능하다" |
| **L3** | 핵심 원칙 | FP1-FP6 | "사람의 판단만이 영속 자산이다", "번역은 규칙 기반이다" |
| **L4** | 설계 판단 | DJ1-DJ10, JDD 설계 판단 | "design.md 확장", "캐리포워드 생명주기 관리" |
| **L5** | 구현 재기술 | 파이프라인, 산출물, 도구 | Crystallize, Delta Manifest, Translation Rules |

### L5에 대한 주의

L5는 기존 구현의 **재기술(re-description)**이다. Translation Ontology는 새로운 구현을 추가하지 않는다 — 기존 파이프라인이 이미 수행하고 있는 연산을 번역 프레이밍으로 재해석한다. 이 재해석은 "구조적 유비(structural analogy)"다. 수학적 대응(미적분, 사영)은 동형(isomorphism)이 아니라 유비로서 유효하다 — 동일한 연산을 다른 언어로 기술하되, 완전한 수학적 등가는 주장하지 않는다.

## 4. 도출 경로

### 4.1 JDD의 도출 (A2 → FP1 → 매체(H7) → JP)

```
A2: 방향은 사람만이 정한다
  → FP1: 사람의 판단만이 영속 자산이다
    → H7: 재생성 비용이 감당 가능하다
      → 설계 판단: 산출물은 소모품이며 재생성 가능
        → H3: 사용자는 자신의 문법으로만 정확성을 검증
          → 매체로서의 산출물: 구체적 산출물 위에서 판단
            → JP1 (방향 검증) + JP2 (목표 검증)
```

**JP1의 고유 기능 — 방향 검증(direction validation)**:

JP1은 단순히 "AI가 느리니까 중간에 체크하자"가 아니다. JP1에는 JP2가 대체할 수 없는 고유 기능이 있다:

| 검증 유형 | JP1 (방향 검증) | JP2 (목표 검증) |
|-----------|----------------|----------------|
| "빠진 시나리오가 있는가?" | **필수** — 프로토타입에 없는 것은 보이지 않음 | 부분적 — 존재하는 것만 검증 가능 |
| "우선순위가 맞는가?" | **필수** — 요구사항 수준에서만 판단 가능 | 불가 — 프로토타입은 우선순위를 표현하지 않음 |
| "이 시나리오가 실제 고객과 맞는가?" | **필수** — 고객 여정 서사로 제시 | 부분적 — 구현된 것만 체험 가능 |
| "화면이 자연스러운가?" | 불가 — 텍스트 수준 | **필수** — 직접 체험 |
| "기능이 기대대로 동작하는가?" | 불가 — 기술 명세 수준 | **필수** — 직접 조작 |

따라서 JP1은 AI 속도가 무한히 빨라져도 완전히 사라지지 않을 수 있다. 다만 형태가 변할 수 있다 — 예를 들어, 프로토타입과 함께 요구사항 체크리스트를 제시하는 형태로.

### 4.2 DDD의 도출 (A1+H1+H2 → 2문법 → FP6(H4) → FP4)

```
A1: 핵심 연산은 번역이다
  + H1: 두 문법은 분리 가능하다
  + H2: 결정론적 매핑 규칙이 존재한다
    → 2문법 모델: User Grammar ↔ Development Grammar
      + H4: 소실 정보는 별도 출처에서 복원 가능
        → FP6: 번역은 규칙 기반이다
          → Complete Specs = translate(Prototype) + carry-forward
            → FP4: 델타 정의가 주요 목표
              → Delta = Complete Specs - Brownfield
```

**FP6 다중 입력 구조**:

번역의 입력은 프로토타입 하나가 아니다. 완전한 번역은 세 가지 입력을 결합한다:

| 입력 | 역할 | 출처 |
|------|------|------|
| **프로토타입** | 사용자 문법으로 표현된 목표 상태 | JP2 승인 |
| **캐리포워드 출처** | 비가시 요구사항 (NFR, 보안, 마이그레이션) | PRD, Architecture, Brownfield |
| **Brownfield 베이스라인** | 현재 시스템 상태 | brownfield-context.md L1-L4 |

```
translate(Prototype, CarryForwardSources, Brownfield)
  = translate(Prototype) + carry-forward(PRD, Architecture, Brownfield) - Brownfield
  = Delta
```

세 입력 중 하나라도 누락되면 번역은 불완전하다:
- 프로토타입 없이: 번역 대상이 없음
- 캐리포워드 없이: 적분상수 누락 (비가시 요구사항 소실)
- Brownfield 없이: 델타를 계산할 수 없음 (Greenfield에서는 ∅로 퇴화)

### 4.3 경제적 원칙의 도출 (H5+H6 → 입력 품질, 재생성, 비용)

```
H5: AI 생성이 리뷰 가능하다
  + H6: 사전 입력이 품질을 높인다
    → 입력이 반복을 줄인다
      + H7: 재생성 비용이 감당 가능하다
        → 수정보다 재생성 (FP5)
          → 비용 공식: C_total = C_input + C_gen × N_gen + C_judge × N_judge + C_carry + C_brownfield
```

경제적 원칙은 전적으로 경험적 전제(H5-H7)에 의존한다. 이 전제들이 변하면:
- H5 깨짐 → 산출물이 리뷰 불가 → 시스템 전체가 작동하지 않음
- H6 깨짐 → inputs/가 무의미 → 순수 반복(iterate) 모델로 전환
- H7 급감 → 재생성 비용 ≈ 0 → JP1 불필요, Brief → JP2 직행

## 5. 문서 관계와 구조적 위치

### 4개 문서 책임 매트릭스

| 관심사 | JDD | DDD | Blueprint | Translation Ontology |
|--------|-----|-----|-----------|---------------------|
| **왜 사람의 판단이 필요한가** | 주 | - | 요약 | 도출 경로 |
| **두 문법과 번역이란 무엇인가** | 참조 | 주 | 요약 | 전제 구조 |
| **파이프라인은 어떻게 작동하는가** | - | 주 | 주 | 재기술 |
| **사용자는 무엇을 하는가** | 요약 | - | 주 | - |
| **원칙들은 어디서 오는가** | 선언 | 선언 | 선언 | **도출 경로** |
| **전제가 깨지면 무엇이 변하는가** | - | - | 리스크 모델 | **전제 체계** |

Translation Ontology는 다른 문서의 내용을 **반복하지 않는다**. 각 문서가 선언하는 원칙들이 어디서 도출되는지, 그리고 어떤 전제 위에 서 있는지를 밝히는 것이 이 문서의 역할이다.

### Sprint Kit과 BMad의 관계

**현재 상태**: Sprint Kit은 BMad Method를 활용하며, 현재 BMad에 종속되어 있다.

구체적 종속 지점:
- `auto-sprint.md`: BMad 에이전트 경로 하드코딩 (Mary, John, Winston)
- `crystallize.md`: BMad 에이전트 경로 하드코딩 (John, Winston)
- `jdd-sprint-protocol.md`: `_bmad/docs/prd-format-guide.md` 필수 참조
- 포맷 가이드: BMad 내부(`_bmad/docs/`)에 위치

**이론적 구조**: Sprint Kit의 논리적 핵심(번역, 델타, 판단)은 BMad에 의존하지 않는다. BMad가 담당하는 것은 "답 발견" Phase(Phase 1)의 탐색 품질이며, 이는 다른 프레임워크(GSD 등)로 대체 가능하다. 접점은 산출물 포맷 계약(planning-artifacts/ 디렉토리)이다.

**분리 로드맵**: [`reviews/translation-ontology-roadmap.md`](reviews/translation-ontology-roadmap.md) §1 참조.

## 6. 교정 사항

Translation Ontology 논의 과정에서 기존 문서의 서술을 교정해야 할 사항들이 발견되었다.

### 6.1 JP1: 방향 검증 고유 기능

**이전 서술**: "JP2가 본질적인 판단 시점이며, JP1은 현재 기술 한계에 대한 실용적 보완이다."

**교정**: JP1에는 JP2가 대체할 수 없는 **방향 검증(direction validation)** 고유 기능이 있다. 프로토타입에 없는 시나리오의 누락 감지, 요구사항 수준의 우선순위 판단, 고객 여정과의 정합성 확인은 JP2에서 수행할 수 없다. JP1은 기술 한계에 대한 보완인 동시에, 방향 검증이라는 고유 역할을 가진다.

**교정 범위**: JDD line 149, Blueprint lines 772-774. (§4.1 JP1 테이블 참조)

### 6.2 FP6: 다중 입력 번역 규칙

**이전 서술**: 번역 입력이 프로토타입 하나인 것처럼 서술.

**교정**: 완전한 번역은 프로토타입, 캐리포워드 출처, Brownfield 베이스라인의 세 가지 입력을 결합한다. 프로토타입만으로는 적분상수(캐리포워드)가 누락되어 번역이 불완전하다. (§4.2 FP6 다중 입력 구조 참조)

**교정 범위**: DDD FP6 뒤에 다중 입력 구조 설명 추가.

### 6.3 재생성: 출처(provenance) 기준

**이전 서술**: 재생성 범위가 "피드백 규모"에 따라 결정된다고만 서술.

**교정**: 재생성 범위는 피드백의 **출처(provenance)**에 따라 더 정교하게 결정된다:

| 피드백 출처 | 의미 | 재생성 범위 |
|------------|------|------------|
| 사용자 입력 원문에 이미 있었음 | 번역/생성 오류 | 해당 단계만 재생성 |
| 사용자 입력에 없었으나 입력에서 추론 가능 | AI 추론 누락 | 해당 단계 + 하류 재생성 |
| 사용자 입력에 전혀 없는 새로운 요구 | 새로운 입력 | Brief 수정 후 파이프라인 재시작 |

출처 기반 분류가 재생성 비용을 더 정밀하게 추정할 수 있게 한다.

**교정 범위**: JDD "수정보다 재생성" 섹션 뒤에 추가.

### 6.4 비용 공식 정교화

**이전 서술**: `총 비용 = (사전 입력 비용) + (생성 비용 × 생성 횟수) + (판단 비용 × 판단 횟수)`

**교정**: 3항 공식은 직관적이지만, 캐리포워드 비용과 Brownfield 수집 비용이 누락되어 있다:

```
C_total = C_input + C_gen × N_gen + C_judge × N_judge + C_carry + C_brownfield
```

| 항 | 의미 | 줄이는 방법 |
|----|------|------------|
| C_input | 사전 입력 수집/정리 비용 | 기존 자료 활용 |
| C_gen × N_gen | AI 생성 비용 × 반복 횟수 | C_input ↑ → N_gen ↓ |
| C_judge × N_judge | 판단 비용 × 판단 횟수 | C_input ↑ → N_judge ↓ |
| C_carry | 캐리포워드 등록/검증 비용 | 구조화된 PRD/Architecture |
| C_brownfield | Brownfield 수집/파싱 비용 | MCP/--add-dir 자동 수집 |

3항 공식은 사용자 대면 설명에서 충분히 유효하다. 5항 공식은 시스템 내부 최적화에 사용된다.

**교정 범위**: Blueprint S4.4에 5항 공식 병기.

### 6.5 BMad: "확장팩" → "활용" (현재 종속 인정)

**이전 서술**: "Sprint Kit = BMad 실행 확장팩", "BMad가 기반 플랫폼"

**교정**: "확장팩(extension pack)"은 Sprint Kit이 BMad의 설계에 맞추어 구축되었음을 암시한다. 실제로는 Sprint Kit이 BMad를 **활용(utilize)**하되, 논리적 핵심(번역, 델타, 판단)은 BMad에 의존하지 않는다. 현재 구현은 BMad에 종속되어 있으며, 이 종속의 분리는 향후 과제다.

- "BMad 실행 확장팩" → "BMad Method를 활용"
- "BMad가 기반 플랫폼" → "BMad에 현재 종속, 분리는 로드맵 참조"

**교정 범위**: JDD lines 3, 17-53; Blueprint lines 250, 888, 901.

## 7. 한계와 열린 질문

### 7.1 "번역"으로 설명되지 않는 활동

Sprint Kit 파이프라인에는 번역 프레이밍으로 포착되지 않는 활동이 있다:

- **탐색(Phase 1)**: PRD 생성, Architecture 결정은 "번역 대상을 찾는" 활동이지 번역 자체가 아니다. BMad 에이전트의 퍼실리테이션, Party Mode 분석 등은 번역과 무관한 발견 과정이다.
- **구현(Phase 3)**: Worker가 델타를 코드로 변환하는 것은 번역이라기보다 "실현(realization)"이다. 코드 생성에는 번역 규칙 외에 구현 판단(라이브러리 선택, 성능 최적화 등)이 필요하다.
- **검증(Validate)**: Judge 에이전트의 품질/보안/비즈니스 검증은 번역 정확성 검증이 아니라 독립적인 품질 보증이다.

### 7.2 "두 문법" vs "N-사영"의 존재론적 불명확성

DDD §5에서 "2문법 모델은 위계가 있는 N-사영 모델의 특수 사례"라고 서술한다. 이것은 존재론적 질문을 야기한다:

- **실제로 2개인가 N개인가?** 2문법 모델이 실용적 단순화인지, 본질적 구분인지가 불명확하다.
- **위계는 고정인가?** "고객 사영 = 목적 함수, 나머지 = 제약"이라는 위계는 모든 시스템에 적용되는가? (예: 보안이 목적 함수인 시스템은?)
- **사영 사이의 상호작용**: N-사영 모델에서 각 사영의 캐리포워드를 독립적으로 관리할 수 있는가, 아니면 사영 간 상호작용이 존재하는가?

현재 Sprint Kit은 2문법 모델로 충분히 작동한다. N-사영 확장은 이론적 가능성으로 남겨둔다.

### 7.3 경험적 전제(H5-H7)의 검증 상태

| 전제 | 검증 방법 | 현재 데이터 |
|------|----------|------------|
| H5 (AI 리뷰 가능) | JP1/JP2 Comment 비율 | 제한적 — 소수 Sprint 관찰 |
| H6 (입력이 품질 향상) | Grade A vs C Brief 비교 | 정성적 관찰만 존재 |
| H7 (재생성 비용 감당) | 사이클 소요 시간 측정 | 5-15분/사이클 (현재 관찰) |

이 전제들의 체계적 검증은 Sprint Kit의 실제 사용이 축적되어야 가능하다.

### 논의 경과

Translation Ontology는 DDD의 수학적 프레이밍(§5) 논의에서 파생된 메타 분석이다. 4개 핵심 문서(JDD, DDD, Blueprint, Terminology Map)를 관통하는 통합 프레이밍을 모색하는 과정에서 발견되었다.

주요 논의 단계:
1. **통합 시도**: 4개 문서의 원칙들이 공통 기반에서 도출될 수 있는가?
2. **"번역" 프레이밍 발견**: Crystallize의 본질이 문법 간 번역임을 인식
3. **공리 체계 시도**: "소프트웨어는 번역이다"를 공리로 설정하려는 시도
4. **파티모드 3중 검증**: 논리학자, 편집자, Devil's Advocate에 의한 검증
5. **교정 후 확정**: 과잉 주장 교정 → 근본 관점 + 전제 체계로 재구성

### 파티모드 검증 결과

3명의 검증자가 초기 초안을 검토하여 다음을 교정했다:

| 초기 주장 | 교정 | 검증자 | 이유 |
|-----------|------|--------|------|
| "공리(axiom)" | "근본 관점(foundational perspective)" | 논리학자 | 숨겨진 전제 H1-H7이 있으므로 공리가 아님 |
| "소프트웨어는 번역이다" | "Crystallize는 번역이다" | Devil's Advocate | Phase 1(탐색), Phase 3(실행)은 번역이 아님 |
| "구조적 동형(isomorphism)" | "구조적 유비(structural analogy)" | 논리학자 | 완전한 수학적 등가를 주장하기에는 근거 부족 |
| QUERY-N/TECH-N 설계 포함 | 로드맵으로 분리 | 편집자 | 미구현 설계를 본론에 포함하면 혼란 |
| BMad 분리 방안 포함 | 로드맵으로 분리 | 편집자 | 현재 상태 기술과 향후 과제를 분리 |

이 교정들이 반영된 결과가 이 문서의 §1-§7이다.
