## 1. 배경 및 문제 정의

### 현재 매칭 방식

EduTalk는 1:1 영어/일본어 회화 수업을 랜덤 튜터 매칭으로 제공하고 있음. 수강생이 시간대를 예약하면 시스템이 가용한 튜터를 자동 배정하는 구조.

### 문제

랜덤 매칭 특성상 이전에 부정적 경험이 있었던 튜터와 재매칭될 수 있음. 이 경우 수강생은 수업을 취소하거나 시간대를 변경하게 되며, 이는 만족도 하락과 이탈로 이어짐.

### CS 데이터 (CX매니저 공유)

- 최근 3개월 CS 인입 중 튜터 관련 불만: **약 100건**
- 이 중 "이 튜터와 다시 수업하고 싶지 않다" 유형: **약 18%** (18건)
- 현재 대응: CX팀이 수동으로 매칭 제외 처리 → 운영 공수 발생
- 월평균 수동 제외 처리: 15~20건

**CX매니저**: "수동 처리가 번거로운 것도 있지만, 더 큰 문제는 수강생이 CS에 연락하기 전에 그냥 수업을 취소하거나 이탈하는 경우입니다. CS로 들어오는 건 빙산의 일각이에요."

### 외부 설문 결과 (운영매니저 공유)

- 외부 설문으로 수강생 대상 조사 실시: **약 500명** 응답
- "랜덤 매칭에서 만나고 싶지 않은 튜터를 만난 적 있는가": **56%가 "예"**
- "그때 어떻게 했는가": 수업 취소 (38%), 그냥 수업 진행 (45%), CS 문의 (12%), 기타 (5%)

### NPS/평가 현황

- 현재 수업 후 NPS 평가 응답률: **23%**
- 실제 수업 완료 건 대비 별점을 남기는 비율: **약 6%**
- 평가 데이터가 극히 부족하여 튜터 품질 모니터링이 어려움

**PM**: "평가율 6%는 사실상 모니터링이 안 되는 수준입니다. 이번 기능에서 평가 수집과 차단을 연계하면 두 마리 토끼를 잡을 수 있어요."

## 2. 기능 목적

1. **수강생 만족도 향상**: 원치 않는 튜터와의 재매칭 방지
2. **운영 자동화**: CX팀 수동 제외 처리 → 수강생 셀프서비스로 전환
3. **튜터 모니터링 강화**: 수업 후 평가 데이터 수집률 향상

### 기대 효과

- **LTV 증가**: 불만으로 인한 이탈 감소 → 수강 지속률 향상
- **환불 감소**: 튜터 불만 관련 환불 월 **약 300만원** 감소 예상
- **평가 수집률**: 현재 6% → 목표 20% 이상

## 3. 기능 설계

### 3-1. 수업 후 평가 (Rating)

**트리거 조건**:
- ClassBoard(화상 수업)에서 퇴장 후 EduTalk 예약 탭 진입 시
- 해당 수업에 대해 미평가 상태일 때
- 하루 1회만 노출 (같은 날 여러 수업이 있어도 가장 최근 1건만)

**평가 플로우**:

```
[팝업 노출: "오늘 수업은 어떠셨나요?"]
    ↓
[별점 선택: 1~5점]
    ↓
┌─ 긍정 (3~5점) ──→ [좋았던 점 선택 (복수 선택)]
│                        - 발음이 좋아요
│                        - 대화가 자연스러워요
│                        - 수업 준비가 잘 되어 있어요
│                        - 분위기가 좋아요
│                        - 기타
│                    → [제출] → 완료
│
└─ 부정 (1~2점) ──→ [아쉬운 점 선택 (복수 선택)]
                         - 수업 준비가 부족해요
                         - 시간을 잘 지키지 않아요
                         - 대화가 부자연스러워요
                         - 태도가 불친절해요
                         - 기타
                     → [이 튜터를 다시 만나지 않을래요] 체크박스
                     → [제출] → (체크 시) 차단 확인 팝업 → 완료
```

**저장 로직**:

| 사용자 행동 | 저장되는 데이터 |
|------------|---------------|
| 별점 선택 후 "다음" 클릭 | 별점만 저장 |
| 사유 선택 후 "제출" 클릭 | 별점 + 사유 저장 |
| "건너뛰기" 클릭 | 아무것도 저장 안 함 |
| 팝업 닫기 (X 버튼) | 아무것도 저장 안 함 |

**디자이너**: "건너뛰기는 CTA 버튼이 아니라 텍스트 링크로 처리하는 게 좋겠어요. 평가를 유도하되 강제하지 않는 느낌으로요."

**PM**: "동의합니다. '건너뛰기'는 클릭 가능한 텍스트로, 시각적으로 약하게 처리해주세요."

### 3-2. 튜터 차단 (Blocking)

**진입점 3가지**:
1. **수업 후 평가**: 부정 평가(1~2점) 시 차단 제안
2. **수업 이력 상세**: 과거 수업의 상세 페이지에서 직접 차단
3. **차단 관리 페이지**: 목록에서 추가/해제

**차단 한도**: 언어당 최대 **5명**

**기획리드**: "5명 제한은 매칭 풀 보호를 위한 것입니다. 무제한으로 열면 매칭 실패율이 올라갈 수 있어요."

**한도 초과 시**:
- 6번째 차단 시도 시 안내 메시지: "차단 가능한 튜터 수를 초과했습니다. 차단 관리 페이지에서 기존 차단을 해제한 후 다시 시도해주세요."
- 차단 관리 페이지로 이동 유도

### 3-3. 차단 관리 페이지

**위치**: MyPage > 수업 설정 > 튜터 관리

**표시 정보**:
- 차단된 튜터 프로필 (이름, 사진)
- 차단 일시
- 마지막 수업 일시
- 차단 해제 버튼

**차단 해제 플로우**:
- 해제 버튼 클릭 → 확인 팝업 → 해제 완료
- 해제 후 해당 튜터는 다시 매칭 풀에 포함

**비활성 튜터 처리**:
- 퇴사/장기 비활성 튜터는 목록에는 표시하되 '비활성' 라벨 부착
- 비활성 튜터도 차단 한도에 포함 (사용자가 직접 해제 가능)

**운영매니저**: "비활성 튜터도 한도에 포함하면 사용자 입장에서 답답할 수 있지 않을까요?"

**기획리드**: "처음에는 포함하고, 이후 데이터를 보면서 자동 해제 정책을 검토하기로 해요. MVP에서는 단순하게 갑시다."

## 4. 정책 범위

| 항목 | 정책 |
|------|------|
| 차단 단위 | 사용자 ID 기준 (기기/세션 무관) |
| 차단 한도 | 언어당 5명 (영어 5명 + 일본어 5명 별도) |
| 비활성 튜터 | 한도에 포함, 차단 관리에서 '비활성' 표시 |
| 차단 해제 | 사용자가 직접 해제 가능 |
| 매칭 반영 | 차단 즉시 매칭 풀에서 제외 |
| 데이터 보존 | 해제 후에도 차단 이력은 보존 (관리자 조회용) |
